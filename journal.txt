## 2026-05-14 — Architecture B+: full Gemini-family rebuild with two-model crisis race

### What

Rebuilt every AI call in the codebase around the combined-evaluation bundle
run on 2026-05-14. The bundle was a 108-scenario, 23-variant test (1188 calls
for the combined run + 2484 calls for the Gemini-thinking variants) covering
tool-calling, multi-turn continuity, mental-health response quality, tagger
accuracy, routing classifier behaviour, complex synthesis, and code. Headline
findings that drove the redesign:

- **Gemini 3.1 Pro preview is unusable** for text generation (108/108 errors
  in the combined run, 92-97% errors on the thinking-variants run). It stays
  in the codebase only as the primary transcription model (audio API path
  differs from text), per Roma's explicit instruction.
- **Gemini 2.5 Flash-Lite GA with thinkingBudget 512 was the highest scorer
  in the entire Gemini family** — 73.5/108 overall, 12/15 on mental-health
  scenarios (Cat C), 8.5/9 on complex synthesis (Cat F), 8/9 on code (Cat G).
- **Gemini 2.5 Pro with thinkingBudget 128 was the best Pro variant** on this
  workload (68/108) and is 2.3× faster than Flash-Lite b512. Higher budgets
  (8192, 24576) underperformed b128 on Pro — Pro over-thinks simple decisions
  when given headroom. Counter-intuitive but data-confirmed.
- **Llama 3.3 70B scored 22.5/108 overall** under the Xaridotis persona size
  (~22KB system context). The previous mood-flow tier-1 model was effectively
  broken under our prompts.
- **gpt-oss-120b is excellent on tool-calling but poor on mental health**
  (6/15 Cat C, leaks "crisis line" and reaches for save_episode reflexively).
  Architecture B+ does not use gpt-oss; the persona's tool-discipline rules
  already cover the lane gpt-oss would have helped on.

### Architecture B+ — routing summary

User-facing conversational lanes:
- Main chat:           gemini-2.5-flash-lite, thinkingBudget=-1 (dynamic)
- Crisis turns:        gemini-2.5-pro b128 RACED against gemini-2.5-flash-lite b512
                       via Promise.race — first non-error response wins. Token
                       cost ×2 on flagged turns only (estimated <50/day).
- Deep / cron synth:   gemini-2.5-pro b128 → gemini-2.5-flash-lite b512
- Short responses:     unchanged — Gemma 4 via Cloudflare, then Gemini fallbacks
- Image gen / edit:    unchanged (3 Pro Image → 3.1 Flash Image)

Background tasks (all migrated to Gemini 2.5 GA family):
- Conversation tagger: gemini-3.1-flash-lite-preview, thinkingLevel='minimal'
                       (new Tier 1 — 2571ms avg, 7.5/12 on Cat E)
                       → Llama 3.1 8B (resilience anchor)
                       → Gemma 4 26B (reasoning fallback)
                       → gemini-3-flash-preview
                       → heuristic floor
- Observation extract: gemini-2.5-flash-lite dynamic, Llama 8B fallback
- Mood entry tagger:   gemini-2.5-flash-lite dynamic, Llama 1B fallback
- Reaction interpret:  gemini-2.5-flash-lite dynamic, Llama 8B fallback
- Memory dedup:        gemini-2.5-flash b8192, GLM-4.7 fallback
- Style card consol:   gemini-2.5-flash b8192, GLM-4.7 fallback
- Persona evolution:   gemini-2.5-flash-lite dynamic, Llama 8B fallback
- Mood micro-ack:      gemini-2.5-flash-lite dynamic, Gemma 4 fallback
- Mood synthesis:      4-tier cascade — 2.5 Pro b128 → 2.5 FL b512 → 2.5 Flash b8192 → GLM 4.7
- Response curator:    Hermes 2 Pro (CF) → gemini-2.5-flash-lite dynamic
- Transcription:       gemini-3.1-pro-preview → gemini-3.1-flash-lite-preview

### Two-model crisis race

New function `generateCrisisResponse` in `src/lib/ai/gemini.js` fires 2.5 Pro
b128 AND 2.5 FL b512 in parallel via `Promise.race` (technically
`Promise.any`-style — first to resolve to non-empty text wins). If the first
to return failed, awaits the other arm. If both arms fail, last-resort
fallback is a single 2.5 Pro dynamic call.

The function is exported but **not yet wired into handlers.js**. Wiring is
the explicit next step: the `tagConversationMode` classifier already returns
`mode: 'crisis'` when appropriate, but the current router consumes that
output for routing-tier selection rather than for dispatching to the
dedicated crisis function. That wiring is a separate follow-up patch.

### Gemini-routed background helper

New export `geminiBackgroundGenerate(env, model, prompt, systemPrompt, opts)`
in `src/lib/ai/gemini.js` mirrors `cfAiGenerate`'s shape (returns trimmed
text or null) so background tasks can swap providers without restructuring.
Supports both `opts.thinkingBudget` (number, for 2.5 GA) and
`opts.thinkingLevel` (minimal/low/medium/high, for 3.x preview).

### Risks worth watching after deploy

1. **`googleSearch` + `functionDeclarations` combination on 2.5 GA.** The
   existing `buildConfig` in gemini.js still registers both. The 3.x preview
   docs explicitly note this combination is unsupported, but the 2.5 GA models
   have a different behaviour and Roma's existing production code has this
   already. Watch tail logs for `INVALID_ARGUMENT` errors mentioning tools
   after deploy. If they appear, split into either-or per call.
2. **3.1 Pro transcription error rate.** Audio path may behave differently
   to text, but if Pro errors at the 92-97% rate we saw for text, the FL
   preview fallback would fire on most transcription requests — returning
   to current behaviour with extra 8s latency on every call. Instrument the
   first 48-72 hours via `transcribe_primary_failed_falling_back` counts.
3. **`generateCrisisResponse` is exported but not wired.** Crisis turns
   currently still route through the regular conversational lane. Wiring is
   the next deliberate step; do not deploy expecting crisis routing yet.
4. **Gemini API quota.** Background tasks (mood micro-ack, observation,
   tagger, dedup) now hit Gemini instead of Cloudflare for the happy path.
   Net cost change is roughly neutral (Pro 3.1 was expensive when it worked,
   removing it offsets the upgraded background tier) but quota usage shifts.
   Worth monitoring `429` rates on Gemini for the first week.

### What is NOT done in this patch

- **handlers.js wiring of generateCrisisResponse.** The function is exported
  and ready, but no call site dispatches to it. The next step is one of:
  (a) extend the router rules in `src/ai/router.js` to detect crisis output
  from `tagConversationMode` and route to crisis lane, or (b) wrap
  `createChat`/`sendChatMessageStream` so they detect mode='crisis' from
  context and switch path. Roma to decide which.
- **No env-var toggle for safe rollback.** All files are changed in place.
  Rolling back requires `git revert` of this commit, then redeploy.
- **Cache compatibility audit.** `setupCache` still creates caches with the
  primary model (`gemini-2.5-flash-lite`). Persona+tools size (~14k tokens)
  is above the 4096-token cache minimum so caching should continue to work,
  but the 2.5 GA + cache + googleSearch + functionDeclarations combination
  is new and has not been verified end-to-end.
- **Background-task evaluation bundle.** Some background changes (memory
  dedup, style card consolidation) are inferred from Cat A/F/D performance
  rather than directly tested. A dedicated bundle covering
  summarisation/triple-extraction would tighten confidence before the next
  refactor.

### Lessons worth carrying forward

1. **The model-family ranking changed.** 2.5 GA is now the workhorse, not the
   3.x preview. Counter to the obvious instinct that "newer = better". The
   bundle data was specific to Xaridotis prompts (22KB persona, 34 tools) —
   it should not be assumed to generalise without re-running on the target
   prompts.
2. **Higher thinking budget is often worse.** Pro b128 outperformed b8192 and
   b24576 on this workload. When a model can decide to think more, it tends
   to over-think. Explicit low budgets often beat dynamic.
3. **Two-model race is a defensible architecture for high-stakes lanes.** On
   crisis turns, the latency floor of "first non-error response wins" is
   bounded by the faster model (Pro at 3.7s) while quality is bounded by
   the better model (FL b512 at 12/15 Cat C). The 2× cost penalty applies
   only to flagged turns, which are <50/day. Watch for over-flagging by
   the classifier inflating that count.
4. **Background tasks were silently expensive.** The previous setup was running
   Llama 3.3 70B on every mood entry (broken under persona), and gemini-3.1-pro
   on every cron tick for evolution-related work. The combined cost of
   "cheap CF" tasks was higher than expected because the persona size made
   Llama unreliable enough to fall through to Gemini. Quality and cost both
   improved by going straight to Gemini 2.5 FL where it works.

## 2026-05-13 — Layer 3: workflow as single source of truth for evening check-in

### What

Rewrote the evening mood check-in flow around the Cloudflare Workflow as the
sole orchestrator. Retired the dual-system competition between the workflow
and the KV-based cron safety-net that was causing duplicate keyboards. The
workflow now owns step ordering, timeout handling, message cleanup, and
synthesis. Callbacks and the AI tool path commit to D1 as before (Issue B);
they now also dispatch events to the workflow so it can progress whether the
user taps a keyboard or types.

### Six locked decisions

1. Auto-terminate any existing workflow instance when a new /mood or cron
   evening fires. No dual instances per day.
2. Same workflow class serves both cron evening and manual /mood, gated by a
   `source` payload field and a `respect_quiet_hours` flag.
3. `waitForEvent` calls are wrapped in try/catch. On 4h timeout the workflow
   deletes the active sent message, marks itself aborted, and jumps to the
   synthesis step. No errored workflow corpses.
4. The extreme-score exception is gone. All scores 0–10 now get emotion
   buttons. Clinical concern handling moves into the synthesis text itself.
5. Quiet-hours check sits as the workflow's first step. Manual /mood passes
   `respect_quiet_hours: false`; cron passes `true`.
6. Side effects move into workflow steps where the workflow owns them. Two
   carve-outs (Issue B): the activities keyboard's intermediate toggle UI
   stays in the callback handler, and the D1 commits for activities (Done
   press), photo upload, and AI-tool typed input remain outside the workflow.

### Workflow bridge (B1)

When the user types data instead of tapping keyboards, the AI tool
`log_mood_entry` now dispatches matching workflow events for score,
emotions, activities, and sleep. Without this bridge, the workflow would sit
at `waitForEvent` for the full 4h timeout on any typed input. The bridge
falls through to the legacy `maybeFireSynthesis` queue path only when no
workflow is active for the day (e.g. workflow create failed at cron time).

### Files changed

- `src/workflows/moodEveningCheckin.js` — full rewrite (~611 lines). 13 steps
  with idempotency markers + step.do caching. Every `waitForEvent` wrapped in
  try/catch. Step 13 (`fire-synthesis`) always runs, even after timeout abort.
- `src/bot/handlers.js` — /mood handler auto-terminates old instance and
  creates a new one with `respect_quiet_hours: false`. `mood_emo_done`,
  `handleSleepCallback`, photo upload, and photo skip handlers dispatch events
  to the workflow when active and skip legacy side effects.
- `src/index.js` — `handleMoodPollAnswer` dispatches `score_received` event
  when workflow active. `enqueueHealthTasks` evening branch creates a workflow
  instance instead of enqueueing a `mood_poll` task. The entire
  `mood_flow_safety_net` enqueue block is removed. Morning/midday branches
  get a separate 10-min in-flight lock + 24h done flag (with `today` in task
  body for cross-midnight safety). The queue consumer's `mood_synthesis_final`
  and `mood_flow_safety_net` branches collapse to a single drain branch that
  logs and acks any pre-deploy queued tasks.
- `src/tools/mood.js` — Layer 3 workflow bridge in `log_mood_entry`. When a
  workflow is active for today, dispatch matching events for whichever fields
  were typed (score / emotions / activities / sleep). Skip the legacy
  `maybeFireSynthesis` path on the workflow branch.
- `src/lib/moodFlow.js` — demoted to a write-through cache. Header comment
  updated to document Layer 3 status.

### New log lines to watch in `wrangler tail`

- `workflow_evening_created` — cron or /mood created an instance
- `workflow_terminated_old` — auto-terminate of previous instance
- `workflow_timeout_cleanup` — waitForEvent hit 4h timeout, cleaned up and
  jumped to synthesis
- `workflow_quiet_skip` — cron suppressed by quiet hours
- `mood_workflow_event_sent` — callback dispatched an event to the workflow
- `mood_workflow_send_event_failed` — dispatch failed, fell back to legacy
- `queue_legacy_mood_task_dropped` — pre-deploy queued task drained silently
- `📨 Workflow bridge: <event>` — AI tool path dispatched a typed-input event

### Known deploy hazard

In-flight workflow instances created before deploy are running the OLD
workflow logic (thin coordinator pattern). New callbacks short-circuit and
dispatch events instead of doing the side effects directly. Old workflow
code expects callbacks to do the work, so events arriving at old instances
will not trigger D1 save / ack / emotion buttons. Result: any check-in
mid-stream at deploy time will stall until the user runs /mood to start a
fresh instance.

Mitigation options:
- A) Manually terminate in-flight instances via
  `wrangler workflows instances list xaridotis-mood-eve-checkin --status=running`
  then `wrangler workflows instances terminate xaridotis-mood-eve-checkin <id>`.
- B) Accept risk: typically 0–1 in-flight instances; max 4h timeout; user
  /mood restart is a clean recovery.

Picked B with a quick check via option A pre-deploy.

### How to deploy

```
npx wrangler deploy --minify
git add -A && git commit -m "layer-3: workflow as single source of truth for evening flow" && git push
```

Then tail for `workflow_evening_created` at the next cron evening or after
running /mood.

### Rollback path

Revert the commit and redeploy. The workflow class binding stays bound;
legacy callbacks still handle their D1 commits correctly without the bridge.
The queue branches we removed (`mood_synthesis_final`, `mood_flow_safety_net`)
would need to come back from git history if we want the safety-net behaviour
restored.

### Why I shared the classifier picture before starting

Roma asked at the top of the session whether we have a proper LLM classifier
or just regex routing. The answer is BOTH, and the split is deliberate:

`tagConversationMode` in `src/services/cfAi.js` IS a real 4-tier waterfall
classifier (Llama 3.1 8B → Gemma 4 26B → Gemini Flash → Flash-Lite →
heuristic floor) returning `{mode, confidence, source}` where mode is
venting/processing/transactional/crisis.

But per `src/ai/router.js` rule 8 comment, we ONLY honour two of those four
modes for routing: `crisis` (→ Pro for safety) and `transactional` (→
Flash-Lite for de-escalation). `venting` and `processing` are explicitly
ignored after the 2026-05-04 incident where a casual 30-char message got
tagged 'venting' and routed to Pro at 35s. The bulk of routing is regex
via `COMPLEXITY_PATTERNS.emotional`, `detectComplexTask`, `isSimpleMessage`.

Defensible design: the tagger drives the two high-stakes corners where
false-negatives are expensive (crisis missed, transactional escalated by
mistake); regex handles the volume where false-positives are cheap.

No change needed in this batch — just flagging the picture for the next
session on routing.

## 2026-05-11 (latest) — Mood flow Round 5: Cloudflare Workflows rewrite of /mood evening check-in, + Computing leak fix

### Two changes in one session, both shipped

1. Removed the `codeExecution` Gemini tool at SDK level, killing the long-running `⚙️ Computing...` leak that appeared after reminders and other tool calls.
2. Replaced the KV-state-machine driving the `/mood` evening check-in with a Cloudflare Workflow that owns step ordering durably and fires synthesis as the last step. Trial-gated behind `USE_MOOD_WORKFLOW=true`. Legacy `moodFlow.js` KV path still runs in parallel for now.

The Computing fix was straightforward. The Workflows rewrite took three deploys to land cleanly because of two subtle bugs that I owe to writing too much code before validating assumptions. Both bugs and their fixes are documented below.

### Why a workflow at all

The Round 4 KV state machine had a Bug A + Bug B that surfaced on 2026-05-10: activities keyboards were being re-fired at 20-21 minute intervals if the user took longer than `STALL_THRESHOLD_MS` (20 min) to respond. Root cause: `moodFlow.js` had `FLOW_TTL_SECONDS = 30 * 60` and a stall detector that re-triggered the current step. Both were trying to be helpful and ended up duplicating user-visible UI.

We considered three fixes:
- A. Tune the stall threshold and TTL to not overlap
- B. Remove the stall detector
- C. Move /mood to Cloudflare Workflows where step ordering is durable by construction

C won because Roma scoped it that way and because the stall detector was solving the wrong problem (Telegram already delivers callbacks reliably; users just take time). Workflows replace the homemade state machine with a primitive built for exactly this — multi-step flows with arbitrary pauses, retries, and `step.waitForEvent`.

We stayed with Workflows rather than Durable Objects because D1 still owns the mood data and we did not want to migrate that to DO storage. Workflows + D1 is the right combination here.

### Architecture

New file: `src/workflows/moodEveningCheckin.js`. Class `MoodEveningCheckinWorkflow extends WorkflowEntrypoint`. Twelve steps:

1. `send-score-poll` — posts the 0-10 bipolar poll to Telegram via direct fetch. Stashes poll_id → workflow context in KV under `mood_poll_${pollId}` so the existing poll_answer dispatcher can route the response back.
2. `await-score` — waits 4h for event `score_received` payload `{score}`.
3. `await-emotions` — waits 4h for event `emotions_done` payload `{emotions: string[]}`. The existing handler in `index.js` (`handleMoodPollAnswer`) handles the score-ack AI message and emotion category buttons; we did NOT duplicate that into the workflow. Same for the post-emotions ack via `handleMoodPollAnswer`'s sibling `runEmotionsAck` triggered by the `mood_emo_done` callback handler.
4. `send-activities-keyboard` — workflow renders the keyboard directly (this is the bug-A fix: only the workflow can render it now, not the stall detector).
5. `await-activities` — waits 4h for `activities_done` payload `{added, removed}`.
6. `send-photo-prompt` — renders the photo ask. If `hasPhotoLoggedToday` is true, returns `{skipped: true}` and the workflow skips the wait.
7. `await-photo` — gated behind `if (!photoStepResult?.skipped)`. Waits 4h for `photo_done` payload `{skipped, photo_r2_key?}`.
8. `send-sleep-keyboard` — NEW UX: deterministic keyboard `[4h] [5h] [6h] / [7h] [8h] [9h] / [10h] [11h] [12h+] / [Skip]`. Same skip pattern: if `hasSleepLoggedToday`, returns `{skipped: true}`.
9. `await-sleep` — gated behind `if (!sleepStepResult?.skipped)`. Waits 4h for `sleep_logged` payload.
10. `fire-synthesis` — runs the existing 6-tier cascade `runSynthesisCascade` INLINE inside the workflow step. Sets the KV synthesis guard first to avoid double-fire with the legacy queue path. Sends typing indicator then the synthesis message. Cleans up `moodFlow.endFlow` and `health_checkin_active_${chatId}`.

Six wiring points in `handlers.js` and one in `index.js` (`handleMoodPollAnswer`) emit the events. Each is gated on `env.USE_MOOD_WORKFLOW === 'true' && env.MOOD_EVE_WORKFLOW` and looks up the active instance ID from KV (see Bug 2 below). When the workflow is active, the legacy `driveNextDeterministicStep` call is skipped — the workflow renders the next keyboard, not the callback handler.

New callback `handleSleepCallback` added for `mood_sleep|<value>` pattern. Routes to D1 upsert + emits `sleep_logged` event.

New workflow binding in `wrangler.jsonc`: `MOOD_EVE_WORKFLOW` → class `MoodEveningCheckinWorkflow` name `xaridotis-mood-eve-checkin`. New `vars` block with `USE_MOOD_WORKFLOW: "true"`. Export added to `src/index.js`.

### Bug 1: `createBatch` is idempotent on existing instance IDs within retention window

First run worked end-to-end except for stalling at `await-sleep`. We fixed that with the photo/sleep skip guards (commit `d05f9a2`). On the next run, `/mood` started logging `mood_command_workflow_started` but then nothing else fired — no poll, no Telegram messages, no further workflow events.

Diagnosis took longer than it should have. The instance ID was `mood_eve_${chatId}_${today}` — fully deterministic, one per day. The Cloudflare docs note that `createBatch` is idempotent: "if an existing instance with the same ID is still within its retention limit, it will be skipped and excluded from the returned array." I had read that but underweighted it. The first run's stuck instance (which we terminated via the dashboard) was still in retention, so every subsequent `/mood` silently no-op'd at the `createBatch` call. The `mood_command_workflow_started` log fired regardless because it sits BEFORE the actual idempotency check — misleading. Lesson: log AFTER the side effect, not before.

Quick fix: appended `Date.now()` to the instance ID so each `/mood` invocation is unique. Stashed the instance ID in KV under `mood_workflow_active_${chatId}_${today}` (24h TTL) so the five callback handlers could look it up without reconstructing it.

### Bug 2: `handleMoodPollAnswer` in `index.js` still constructed the old deterministic ID

After the timestamp-suffix fix, the next run got further: `mood_workflow_event_sent` fired for `emotions_done`, `activities_done`, `photo_done`. But `score_received` never appeared. The workflow stalled at `await-score`.

Reason: I had updated the five callback sites in `handlers.js` to read the instance ID from KV, but missed `handleMoodPollAnswer` in `index.js`. It was still constructing `mood_eve_${chatId}_${today}` (no timestamp), calling `MOOD_EVE_WORKFLOW.get(instanceId)`, and the `sendEvent` was being dropped because the instance didn't exist. The catch block was silent — no log line, no surfaced error. The workflow processed `emotions_done` and later events but they were dropped on the floor because the workflow was still in `await-score` (Workflows discard events that arrive before the matching `waitForEvent` is reached).

Fixed by updating `index.js` to read from the same KV key. Lesson: when refactoring the call pattern at one site, grep for every site sharing the same shape.

1. **Workflows `createBatch` is idempotent on terminated instances within retention.** Re-using a deterministic ID across the day will silently no-op once the first instance lands in retention, even if it terminated. 5. **Trial-gating with a feature flag was worth it.** Three deploys, three bugs caught and fixed without users (just me) noticing because the legacy KV path ran in parallel throughout. `USE_MOOD_WORKFLOW=false` would have been a one-line revert.

## 2026-05-07 — Mood flow Round 4: activities and photo always asked, sleep is the only conditional step

### Session arc (important context)

This session went through TWO designs. The first attempt implemented day-level skipping for activities and photo ("if today already has activities, don't ask again"), reusing `computeDayCompleteness` from Round 3 to gate both the flow router and the synthesis trigger. Roma pushed back hard mid-session: "There is a fundamentally wrong implementation. I have never asked not to trigger activities; we even implemented the option to select previously selected activities from previous or current mood checks to track them and change them during the day."

The correct rule is below. The journal entry describes the FINAL design only — the abandoned first attempt is not preserved here because it was never deployed beyond a partial test, but the lesson worth carrying forward is: **the activities and photo keyboards are review-and-modify primitives. Their existence in today's data is not a reason to skip them. Sleep is different because the prose ask creates real friction when repeated.**

### What

Round 3 had a bug. The end-of-flow synthesis used `computeDayCompleteness()` to decide whether the day was "done", and the deterministic flow router used the same day-level check to decide which keyboard or prose ask to send next. That meant: if today already had activities (logged earlier in the day), `/mood` would skip the activities step. If today already had a photo, `/mood` would skip the photo step. The user explicitly never asked for that and pushed back.

The correction: activities and photo are ALWAYS asked. The keyboards already support reviewing today's existing selections and adding or removing items, so re-prompting is the right behaviour. Sleep is the ONLY step that gets skipped if today's data already has it, because asking "how many hours did you sleep last night" twice in one day is real friction.

### Decision: per-step routing, not day-level skipping

`driveNextDeterministicStep` now takes an explicit `step` parameter (`'activities' | 'photo' | 'sleep'`) supplied by the caller. The function no longer reads day-level completeness from D1. It just sends what the caller asked for:

- `step: 'activities'` — always sends the activities keyboard via `sendActivitiesKeyboardTool`. Preamble mentions "ones you have already logged are still here, you can add or remove" so the user knows the keyboard is review-and-modify. Returns true.
- `step: 'photo'` — always sends the photo prompt with a Skip button. Bypasses `sendPhotoRequestTool` (which noops if a photo already exists today) and sends the keyboard directly via `telegram.sendMessage` + `moodFlow.initPhotoPending`. Returns true.
- `step: 'sleep'` — the only conditional step. Calls `moodStore.hasSleepLoggedToday`. If sleep is logged, returns false so the caller can fire `maybeFireSynthesis`. If not, sends the prose "How many hours did you sleep last night?" ask and returns true.

Call sites:
- `mood_emo_done` (after emotions micro-ack) → `driveNextDeterministicStep(…, 'activities')`
- Activities Done callback → `driveNextDeterministicStep(…, 'photo')`
- Photo skip callback AND photo upload handler in `handleMessage` → `driveNextDeterministicStep(…, 'sleep')`. If returns false (sleep already logged), caller imports and calls `maybeFireSynthesis`. The photo upload handler returns early so the AI generation pipeline doesn't run on top.
- Sleep prose reply → user types hours → `log_mood_entry` tool runs → `maybeFireSynthesis` fires from `tools/mood.js`.

### Synthesis trigger simplified

`maybeFireSynthesis` no longer reads day-level completeness flags. The only check is `hasSleepLoggedToday`. Score, emotions, activities, and photo are guaranteed answered "this run" by call-site invariant: the photo handler is the synthesis trigger, and reaching the photo handler means the user has already walked poll → emotions Done → activities Done → photo step in order.

The old log event `mood_synthesis_not_ready` (which dumped a flag breakdown) is replaced with `mood_synthesis_waiting_for_sleep` — clearer signal, since the only thing the function ever waits for now is sleep.

### D1 dedup non-issue

Separate concern raised: "the database shouldn't accumulate redundant records." After querying D1 directly via the Cloudflare MCP, this isn't a real problem. The `mood_journal` schema enforces unique `(user_id, date, entry_type)`, so each entry_type has at most one row per day. Re-running `/mood` UPDATES the existing evening row in place; it doesn't create duplicates. Photo upload either fills `photo_r2_key` on the existing row or overwrites it. The merge logic for activities only writes when adds or removes are non-empty.

If in future we want multiple distinct evening rows per day (one per `flow.started_at` for example), that's a schema change — not in scope.

### What changed

- `src/services/moodSynthesis.js`:
  - `maybeFireSynthesis` rewritten: only checks sleep day-level. Removed `computeDayCompleteness` call and the flags-breakdown log.
  - File header comment updated to reflect new behaviour (no more "day-level completeness reads ALL of today's mood rows" claim).
  - `computeDayCompleteness` function removed entirely (~30 lines + docstring) since `maybeFireSynthesis` was its only caller. Confirmed no other references in `src/`.
  - Trailing duplicate `} catch` block (left over from an earlier edit) cleaned up.
  - Renamed log event `mood_synthesis_not_ready` → `mood_synthesis_waiting_for_sleep`.
- `src/bot/handlers.js`:
  - `driveNextDeterministicStep(env, chatId, userId, threadId, step)` rewritten with explicit step parameter and three branches as described above.
  - `mood_emo_done` calls it with `'activities'`.
  - Activities Done handler calls it with `'photo'` after merge + `advanceStage`.
  - `handlePhotoSkipCallback` calls it with `'sleep'`. Returns false → imports `maybeFireSynthesis` from `moodSynthesis` and fires.

## 2026-05-06 — Mood flow Round 3: stripped mid-flow AI, end-only synthesis on Cloudflare-first cascade

### What

Live test on 2026-05-06 13:05 showed the mood synthesis hanging 127s+128s+128s on Gemini Pro 524s, retrying 3x then falling through to fixed text. Round 1 (queue migration) and Round 2 (stage tracking) had landed but couldn't help — the upstream Pro hangs were the bottleneck.

Round 3 redesign rebuilt the mood flow around two principles: **AI only at the end**, and **Cloudflare-first** with Gemini as fallback.

### Decision: end-only synthesis with day-level completeness

The entire mid-flow AI surface is gone:
- Score-tap acknowledgement: was Pro+thinking (10-25s, 127s on bad days). Replaced with a 2-tier Cloudflare-only micro-ack (Llama 3.3 70B fp8-fast → Gemma 3 12B → static text). 1-3s realistic latency.
- Emotions-Done acknowledgement: was Pro+high-thinking via the queue. Replaced with the same 2-tier micro-ack.
- Activities-Done kick: was a synthetic `handleMessage` call to nudge the AI to call the next tool. Replaced with `driveNextDeterministicStep()` which reads what's missing from D1 and sends the right keyboard or prose ask. Zero AI involved in flow routing.
- Photo-skip kick: same as activities-done.

End-of-flow synthesis is now the ONLY heavy AI call per check-in. Fired exactly once when ALL pieces are in: score, emotions, activities, sleep, photo-or-skipped — all checked across ALL of today's mood rows (morning + midday + evening combined). The cascade is 6 tiers:

1. `@cf/meta/llama-3.3-70b-instruct-fp8-fast` (15s)
2. `@cf/google/gemma-3-12b-it` (15s)
3. `gemini-3.1-flash-lite-preview` (20s)
4. `gemini-3-flash-preview` (25s)
5. `gemini-3.1-pro-preview` (30s)
6. Fixed warm fallback text

Mixed error-feedback + timeout per tier. Each tier wrapped in `Promise.race` so silent hangs (the 524 pattern) bail at the timeout cap. Realistic case: Tier 1 succeeds in 5-8s. Worst case before fixed text: 105s.

### Why "day-level" matters

The day-level completeness check means if sleep was logged in the morning entry (via `log_mood_entry` from a chat reply), the evening flow doesn't ask again. Same for emotions and photo. The synthesis sees the combined picture of the day, not just the evening row. Implemented via `moodStore.getDayEntries(env, userId, date)` which returns all rows for the date; `computeDayCompleteness()` walks them and OR's each piece.

### Prompt style: situation + role, no instructions

All three new prompts (score-ack, emotions-ack, synthesis) use the "describe the situation, set the role, get out of the way" style. No "max N words", no "do NOT mention X", no examples. Just data + "Briefly respond as a supportive and understanding friend who notices." The persona system prompt handles tone and severity.

Corrections applied during prompt design:
- Full poll option text in the score scale (not abbreviated buckets), so the model sees the same wording the user did.
- Dynamic `${period}` ("this morning" / "this afternoon" / "this evening" / "tonight") computed from `flow.started_at` via new `getCheckinTiming()` helper in `moodFlow.js`. Works for cron-triggered AND manual `/mood` at any time of day.
- Removed the word "clinical" from prompts — "His patterns and known triggers from past sessions" reads more like a friend's recall than a chart review.
- Helpline trio (Samaritans 116 123, SHOUT text 85258, NHS 111) only in the end-of-flow synthesis prompt, conditional on safety concern in the data.

- `src/services/moodMicroAck.js` (~280 lines) — score-ack + emotions-ack runners, 2-tier Cloudflare cascade, mixed error+timeout, prompt builders.
- `src/services/moodSynthesis.js` (~330 lines) — 6-tier cascade, day-level completeness checker, `maybeFireSynthesis()` helper with idempotency guard (`mood_synthesis_fired_${chatId}_${date}` KV key, 26h TTL).

## 2026-05-06 (latest+2) — Mood flow Plan A: full queue migration

### What

The kick patch (entry above) was correct in mechanism but doomed by
architecture. Two heavy generations chained back-to-back inside one
`waitUntil` window cannot fit the 30s post-response budget:

- Pro+high-thinking synthesis: 10-25s
- Full `handleMessage` pipeline (tagger, memory, route, cache, generation):
  another 5-15s
- Telegram callback responded in <1s, so the budget started counting from
  the moment the synthesis began

Result: synthesis sent, kick started, generation cancelled mid-stream.
User saw synthesis, then nothing.

Project notes already explicitly flag this constraint:

> `ctx.waitUntil` ceiling is 30 seconds post-response (not configurable
> `cpu_ms`). Media/async work must use Queues (15-min wall clock, 3 retries)
> for reliability.

I missed applying that pattern to the new emotion-flow chain and got
bitten by it. Roma asked for the queue migration explicitly: "all the
mood check steps should be on the queue".

### Architecture decision: Plan A (queue) vs Plan B (Cloudflare Workflow)

Weighed two architectures. Plan A: each heavy step becomes a `TASK_QUEUE`
message, matching the existing `mood_poll_answer` pattern. ~30 min of work,
low risk, no architectural lock-in. Plan B: full Cloudflare Workflow
rewrite (already on the project's roadmap as a dedicated session).

Chose A. Rationale:
- Bug is a `waitUntil` exhaustion; queue solves that. Workflow solves it
  AND many other things we didn't ask for.
- Mid-debug refactor is the wrong time to switch architectures.
- Workflow design needs answers to non-trivial questions (one workflow
  per check-in vs per user, how Telegram callbacks wake a running
  instance, state size, idempotency) — those belong in a dedicated
  design session, not a panic patch.
- Roma's project notes already define Workflows as "agreed for a
  dedicated session" — keeping that scope for later.

### What changed

**`src/bot/handlers.js` — `mood_emo_done` callback handler (~line 2486):**

From 110 lines (synthesis + history + kick) down to ~30 lines. Now does:
1. Edit message markup
2. Save emotions to D1 (`upsertEntry`)
3. Fire-and-forget tagger (CF AI, free, ~500ms)
4. Clear emo selected KV
5. Enqueue `mood_synthesis_after_emotions` to TASK_QUEUE
6. Return

Total inline time: <1s. No Pro calls, no `handleMessage`, nothing that
could stall the `waitUntil` budget.

Fallback path: if `env.TASK_QUEUE` is missing or send fails, post a brief
acknowledgement so the user isn't left staring at a dead chat. The deep
synthesis is sacrificed in that edge case but the flow continues.

**`src/index.js` — queue consumer (added 2 cases after `mood_poll_answer`):**

1. `mood_synthesis_after_emotions`:
   - Re-derives all context fresh from D1/KV/Vectorize (don't trust
     cached state — the queue could fire minutes later in theory).
     Pulls: today's entry, recent 30-day mood history, therapeutic notes,
     semantic vector context, relevant past episodes by emotion.
   - Builds the same therapeutic synthesis prompt as before.
   - Calls `generateDeepResponse` with `thinkingLevel: 'high'`,
     `maxOutputTokens: 2000` (Pro + high thinking, matching Eukara's pattern).
   - Sends synthesis to Telegram.
   - Persists synthetic user turn + synthesis in chat history KV.
   - Enqueues `mood_kick_after_emotions` as a SEPARATE message so the
     synthesis success can't be rolled back by a kick failure.
   - On final retry (attempt 3) failure: sends a brief acknowledgement so
     the user isn't hanging. Throws to retry queue otherwise.

2. `mood_kick_after_emotions`:
   - Builds the same synthetic kick message as before.
   - Awaits `handleMessage(kickMsg, env)` — fresh 15-min budget.
   - On failure: don't notify user (synthesis already landed). Re-throw
     for queue retry. After 3 retries the user can simply type to resume
     the flow naturally.

### Why two queue messages instead of one

Idempotency. If synthesis succeeds and kick fails, retrying ONLY the kick
is safe — synthesis was already sent to Telegram, no duplicate. If we put
both in one message, retrying after kick failure would re-run synthesis
and send a duplicate to the user.

Each message also gets its own 15-min budget and 3-retry budget. If Pro
is slow on synthesis (e.g. 4-min preview overload), the kick still has a
full 15 min when its turn comes.

### Risks acknowledged

- **Latency between synthesis and kick:** queue dispatch adds ~1-3s.
  Acceptable; user is reading the synthesis during this window.
- **Queue failure:** the inline fallback in `mood_emo_done` covers
  TASK_QUEUE missing or send failing. The synthesis fallback covers
  Pro generation failing. The kick has no user-facing fallback because
  the synthesis is already there — user can type to resume.
- **Context staleness:** queue context is re-derived fresh. The only
  thing passed via the queue message is `selected` (the emotion array)
  — everything else comes from D1/KV/Vectorize at queue-handler time.
- **Existing `mood_flow_safety_net` queue task is now redundant** for the
  immediate-after-emotions case (we no longer rely on the AI noticing the
  flow on its own). It still serves as a safety net for genuinely stalled
  flows (e.g. user starts then walks away). Leaving it in place.

### Problem 1: codeExecution leaking Python tracebacks

### Problem 4: Telegram HTML parse failures from model-emitted unknown tags

User flagged the `<b>Evening Affirmations</b>` rendering and asked whether
formatting was Telegram-compatible end-to-end. Investigation also surfaced
the pre-existing `Bad Request: can't parse entities: Unsupported start tag
"string"` error from earlier in the day.

Root cause: `sanitizeTelegramHTML` in `src/lib/formatter.js` used a deny-list
of known-bad tags (`p`, `div`, `br`, `h1-6`, `span` without tg-spoiler, etc.).
Anything the model invented outside that list (`<string>`, `<example>`,
`<note>`, etc.) slipped through to a Telegram parse failure. The strip-all-tags
fallback in `telegram.sendMessage` caught it but users saw degraded output.

Reminder cron path in `src/index.js` already escapes user content correctly
via `escapeHtml` (added in commit `f465422`), so reminders themselves are
not the failure source. The failures come from bot-generated wrappers around
model output (e.g. `<b>Weekly Report</b>\n\n${report}`) where ${report}
contained unexpected tags.

Fix (Option A — allow-list rewrite). Commit `7c724d2`. Rewrote
`sanitizeTelegramHTML` in `src/lib/formatter.js`:
  1. Markdown pre-processing kept (`**bold**`, `__italic__` -> `<b>`/`<i>`)
  2. Allow-list strip: every tag matched against Telegram spec. Plain
     allow-list: b, strong, i, em, u, ins, s, strike, del, tg-spoiler,
     code, pre, blockquote. Attribute-bearing tags handled separately:
     `<a href>` (required href), `<span class=tg-spoiler>` (only that class),
     `<tg-emoji emoji-id=NNN>` (numeric only), `<blockquote expandable>`
     (expandable flag preserved)
  3. Existing balancer (orphan closers / unclosed openers) and whitespace
     collapse retained unchanged
  4. HTML entities (&lt; etc) left untouched so reminder cron's escapeHtml
     output stays escaped

Test coverage: 23 cases ran locally before deploy, all passed. Including the
real failing case (`<string>foo</string>` strips cleanly), pre-escaped
entity preservation, all spec-allowed attribute-bearing tags, balancer
correctness on orphans/unclosed.

Acknowledged trade-off: `<code class="language-python">` attributes are now
stripped. Syntax highlighting in Telegram code blocks no longer works.
Cosmetic only — parse still succeeds. If anyone needs language-hinted code
blocks back, add a special case for `<code>` matching the pre/code spec.
