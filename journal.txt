## 2026-05-04 — Persona tool-routing fix + Flash-Lite fallback

### Problem 1: codeExecution leaking Python tracebacks

Asking "show me my reminders" caused the model to reach for Gemini's built-in
`codeExecution` Python sandbox instead of calling our `list_reminders` tool.
The sandbox tried `default_api.list_reminders()` and `update_schedule()` —
neither exists in the Workers AI Python runtime — leaked NameError tracebacks
as `Result:` lines into the Telegram chat, then confabulated "no reminders
found" despite 5 reminders sitting in D1.

Root cause: `src/lib/ai/gemini.js` enables three tool surfaces simultaneously:
  - `functionDeclarations` (our 32 custom tools)
  - `googleSearch: {}`
  - `codeExecution: {}`

Earlier description sharpening on `list_reminders` and `manage_cloudflare`
(commit 7e6e004) had only redirected the misuse — model switched from
`manage_cloudflare` to the Python sandbox. Tool description quality wins
skirmishes, not the war.

Fix (Option C — soft persona instruction). Commit `39f7b0c`. Added a
`TOOL SELECTION RULES (HARD CONSTRAINTS)` block to
`personas.xaridotis.instruction` covering reminders, mood, memories, episodes,
and therapeutic notes. Persona size grew 15824 → 17274 chars.

Acknowledged: this is a soft constraint and may drift. Escalation path to
Option A (remove `codeExecution: {}` outright in `src/lib/ai/gemini.js` lines
143 + 171) is documented and ready.

Verified working: test "Show me all my reminders" produced
`tool_calls: ["list_reminders"]` and returned all 5 reminders cleanly.

### Problem 2: Flash-Lite mid-stream truncation crashed handler

Second test ("what do I have scheduled") crashed with
`Error: Incomplete JSON segment at the end` from `processStreamResponse_1`
in the `@google/genai` SDK at total_elapsed_ms 11401. SDK threw because
Google ended the stream before a JSON chunk closed.

Two gaps in `handleMessage`'s catch block:
  1. `isOverload` keyword check did not include `"Incomplete JSON segment"`,
     so `isRetryable` came back false and the handler crashed.
  2. The fallback branch only fired when `textModel === PRIMARY_TEXT_MODEL`.
     Flash-Lite had no escape hatch at all.

Fix. Commit `c58d0f4`, deployed `b5e8fe74-9556-4486-8e06-271e25511aeb`.
Two surgical edits to `src/bot/handlers.js`:

  1. Added `isStreamTruncated` flag matching `'Incomplete JSON segment'`,
     folded into `isRetryable`. Kept as a separate boolean so logs can tell
     mid-stream truncation apart from upstream overloads.

  2. Extended fallback condition from `textModel === PRIMARY_TEXT_MODEL` to
     `(PRIMARY_TEXT_MODEL || FLASH_LITE_TEXT_MODEL)`. Both tiers fall to
     Flash. Renamed log event `pro_fallback_to_flash` → `fallback_to_flash`
     with `from`/`to`/`reason` labels for tail diagnostics.

### Considered and rejected

- Adding Gemma 4 26B to the response chain (Gemma → Flash-Lite → Flash → Llama).
  Rejected: Gemma uses OpenAI-compat tool calling, not Gemini
  `functionDeclarations`. Would require translating 32 tool defs and building
  a parallel tool loop — 1-2 day project. Llama 8B can't do reliable 32-tool
  calling at all, so the bottom of the chain was fictional safety. Cost saving
  on transactional turns is sub-pence. Concentrates risk on a model already
  serving as tagger fallback + observation extractor + casual fast-path.

- Flash-Lite-non-streaming as cheaper first retry before Flash. Skipped:
  today's failure was genuine connection truncation (11.4s of generation
  before cut), not throttling. Retrying same model is a coin flip. Flash is
  well-cached and reliable. If recurrence shows non-streaming would help,
  add it then.

### Problem 3: handleArchitectureEvolution running hourly, burning tokens

User flagged that token consumption from architecture deep search was running
high and they had stopped engaging with the resulting interactive PR drafts.

Root cause: `handleArchitectureEvolution` in `src/index.js` had two
contradicting docstrings (`Weekly Deep Search` and `Hourly Deep Search`) and
only checked `londonTime.getMinutes() === 0`. So it fired at the top of every
hour, every day. ~24 runs/day, each making three Gemini calls
(search + read_webpage + PR draft) = ~72 Gemini calls/day plus interactive
PR Telegram messages with approve/dismiss buttons. The schedules.js config
still said Monday 04:00 London but the function never read its own schedule.

Fix (Option B — disable, not restore-to-weekly): user explicitly preferred
disable over restoring weekly cadence. Three changes in commit `6a94778`:
  1. Early return at top of handleArchitectureEvolution with explanatory
     comment and re-enable instructions
  2. Removed `handleArchitectureEvolution(env)` from the queued cron path’s
     `Promise.allSettled` and from the matching names array
  3. Same removal in the fallback cron path and its names array

Reviewed `handleDailyStudy` in same patch (per user request) but did NOT
patch it. Analysis: capped at 1 study/day via `daily_study_<date>` KV lock,
10%-per-hour random gate during 07-23 window. Real cost is 3-4 Gemini
calls/day max. Two orders of magnitude less than architecture_evolution.
Working as intended. Touching it would be scope creep.

To re-enable architecture evolution later: search for
`ARCHITECTURE_EVOLUTION_DISABLED` in `src/index.js`. Recommend gating on
`getSchedule(env, 'architecture_evolution')` with day+hour+minute match before
restoring, so it cannot regress to hourly.

### Problem 4: Telegram HTML parse failures from model-emitted unknown tags

User flagged the `<b>Evening Affirmations</b>` rendering and asked whether
formatting was Telegram-compatible end-to-end. Investigation also surfaced
the pre-existing `Bad Request: can't parse entities: Unsupported start tag
"string"` error from earlier in the day.

Root cause: `sanitizeTelegramHTML` in `src/lib/formatter.js` used a deny-list
of known-bad tags (`p`, `div`, `br`, `h1-6`, `span` without tg-spoiler, etc.).
Anything the model invented outside that list (`<string>`, `<example>`,
`<note>`, etc.) slipped through to a Telegram parse failure. The strip-all-tags
fallback in `telegram.sendMessage` caught it but users saw degraded output.

Reminder cron path in `src/index.js` already escapes user content correctly
via `escapeHtml` (added in commit `f465422`), so reminders themselves are
not the failure source. The failures come from bot-generated wrappers around
model output (e.g. `<b>Weekly Report</b>\n\n${report}`) where ${report}
contained unexpected tags.

Fix (Option A — allow-list rewrite). Commit `7c724d2`. Rewrote
`sanitizeTelegramHTML` in `src/lib/formatter.js`:
  1. Markdown pre-processing kept (`**bold**`, `__italic__` -> `<b>`/`<i>`)
  2. Allow-list strip: every tag matched against Telegram spec. Plain
     allow-list: b, strong, i, em, u, ins, s, strike, del, tg-spoiler,
     code, pre, blockquote. Attribute-bearing tags handled separately:
     `<a href>` (required href), `<span class=tg-spoiler>` (only that class),
     `<tg-emoji emoji-id=NNN>` (numeric only), `<blockquote expandable>`
     (expandable flag preserved)
  3. Existing balancer (orphan closers / unclosed openers) and whitespace
     collapse retained unchanged
  4. HTML entities (&lt; etc) left untouched so reminder cron's escapeHtml
     output stays escaped

Test coverage: 23 cases ran locally before deploy, all passed. Including the
real failing case (`<string>foo</string>` strips cleanly), pre-escaped
entity preservation, all spec-allowed attribute-bearing tags, balancer
correctness on orphans/unclosed.

Acknowledged trade-off: `<code class="language-python">` attributes are now
stripped. Syntax highlighting in Telegram code blocks no longer works.
Cosmetic only — parse still succeeds. If anyone needs language-hinted code
blocks back, add a special case for `<code>` matching the pre/code spec.

Deferred to a follow-up Option C session: centralised
`formatTelegramMessage(template, fields)` helper to make bot-template +
user-data interpolation auto-escape correctly. Currently five sites in
`src/index.js` (Weekly Report, Mid-Week Check-in, Monthly self-assessment,
Things I found interesting, Architecture Deep Search title) interpolate
model output directly into `<b>Title</b>\n\n${msg}` templates. The new
sanitiser catches recurrences but a centralised helper would make this
discipline impossible to forget.
