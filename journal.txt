## 2026-05-07 (latest) — Mood flow Round 4: activities and photo always asked, sleep is the only conditional step

### Session arc (important context)

This session went through TWO designs. The first attempt implemented day-level skipping for activities and photo ("if today already has activities, don't ask again"), reusing `computeDayCompleteness` from Round 3 to gate both the flow router and the synthesis trigger. Roma pushed back hard mid-session: "There is a fundamentally wrong implementation. I have never asked not to trigger activities; we even implemented the option to select previously selected activities from previous or current mood checks to track them and change them during the day."

The correct rule is below. The journal entry describes the FINAL design only — the abandoned first attempt is not preserved here because it was never deployed beyond a partial test, but the lesson worth carrying forward is: **the activities and photo keyboards are review-and-modify primitives. Their existence in today's data is not a reason to skip them. Sleep is different because the prose ask creates real friction when repeated.**

### What

Round 3 had a bug. The end-of-flow synthesis used `computeDayCompleteness()` to decide whether the day was "done", and the deterministic flow router used the same day-level check to decide which keyboard or prose ask to send next. That meant: if today already had activities (logged earlier in the day), `/mood` would skip the activities step. If today already had a photo, `/mood` would skip the photo step. The user explicitly never asked for that and pushed back.

The correction: activities and photo are ALWAYS asked. The keyboards already support reviewing today's existing selections and adding or removing items, so re-prompting is the right behaviour. Sleep is the ONLY step that gets skipped if today's data already has it, because asking "how many hours did you sleep last night" twice in one day is real friction.

### Decision: per-step routing, not day-level skipping

`driveNextDeterministicStep` now takes an explicit `step` parameter (`'activities' | 'photo' | 'sleep'`) supplied by the caller. The function no longer reads day-level completeness from D1. It just sends what the caller asked for:

- `step: 'activities'` — always sends the activities keyboard via `sendActivitiesKeyboardTool`. Preamble mentions "ones you have already logged are still here, you can add or remove" so the user knows the keyboard is review-and-modify. Returns true.
- `step: 'photo'` — always sends the photo prompt with a Skip button. Bypasses `sendPhotoRequestTool` (which noops if a photo already exists today) and sends the keyboard directly via `telegram.sendMessage` + `moodFlow.initPhotoPending`. Returns true.
- `step: 'sleep'` — the only conditional step. Calls `moodStore.hasSleepLoggedToday`. If sleep is logged, returns false so the caller can fire `maybeFireSynthesis`. If not, sends the prose "How many hours did you sleep last night?" ask and returns true.

Call sites:
- `mood_emo_done` (after emotions micro-ack) → `driveNextDeterministicStep(…, 'activities')`
- Activities Done callback → `driveNextDeterministicStep(…, 'photo')`
- Photo skip callback AND photo upload handler in `handleMessage` → `driveNextDeterministicStep(…, 'sleep')`. If returns false (sleep already logged), caller imports and calls `maybeFireSynthesis`. The photo upload handler returns early so the AI generation pipeline doesn't run on top.
- Sleep prose reply → user types hours → `log_mood_entry` tool runs → `maybeFireSynthesis` fires from `tools/mood.js`.

### Synthesis trigger simplified

`maybeFireSynthesis` no longer reads day-level completeness flags. The only check is `hasSleepLoggedToday`. Score, emotions, activities, and photo are guaranteed answered "this run" by call-site invariant: the photo handler is the synthesis trigger, and reaching the photo handler means the user has already walked poll → emotions Done → activities Done → photo step in order.

The old log event `mood_synthesis_not_ready` (which dumped a flag breakdown) is replaced with `mood_synthesis_waiting_for_sleep` — clearer signal, since the only thing the function ever waits for now is sleep.

### D1 dedup non-issue

Separate concern raised: "the database shouldn't accumulate redundant records." After querying D1 directly via the Cloudflare MCP, this isn't a real problem. The `mood_journal` schema enforces unique `(user_id, date, entry_type)`, so each entry_type has at most one row per day. Re-running `/mood` UPDATES the existing evening row in place; it doesn't create duplicates. Photo upload either fills `photo_r2_key` on the existing row or overwrites it. The merge logic for activities only writes when adds or removes are non-empty.

If in future we want multiple distinct evening rows per day (one per `flow.started_at` for example), that's a schema change — not in scope.

### What changed

- `src/services/moodSynthesis.js`:
  - `maybeFireSynthesis` rewritten: only checks sleep day-level. Removed `computeDayCompleteness` call and the flags-breakdown log.
  - File header comment updated to reflect new behaviour (no more "day-level completeness reads ALL of today's mood rows" claim).
  - `computeDayCompleteness` function removed entirely (~30 lines + docstring) since `maybeFireSynthesis` was its only caller. Confirmed no other references in `src/`.
  - Trailing duplicate `} catch` block (left over from an earlier edit) cleaned up.
  - Renamed log event `mood_synthesis_not_ready` → `mood_synthesis_waiting_for_sleep`.
- `src/bot/handlers.js`:
  - `driveNextDeterministicStep(env, chatId, userId, threadId, step)` rewritten with explicit step parameter and three branches as described above.
  - `mood_emo_done` calls it with `'activities'`.
  - Activities Done handler calls it with `'photo'` after merge + `advanceStage`.
  - `handlePhotoSkipCallback` calls it with `'sleep'`. Returns false → imports `maybeFireSynthesis` from `moodSynthesis` and fires.
  - Photo upload handler in `handleMessage` does the same and returns early.

### Verification plan

1. `grep -rn "computeDayCompleteness" src/` — should return nothing.
2. `npx wrangler deploy --dry-run` — typecheck.
3. `npx wrangler deploy --minify`.
4. Live test `/mood`. Tail should show this trace for a day where sleep is already logged from morning:
   - poll tap → score-ack (Llama or Gemma)
   - emotions Done → emotions-ack → `drive_step_activities`
   - activities Done → `drive_step_photo`
   - photo upload or Skip → `drive_step_sleep_skipped` → `mood_synthesis_enqueued` → `mood_synthesis_final` consumer runs → `mood_synthesis_tier_ok`
5. Live test `/mood` for a day where sleep is NOT yet logged:
   - same up to photo step
   - photo upload or Skip → `drive_step_sleep` (prose ask sent)
   - user replies with hours → `log_mood_entry` runs → `mood_synthesis_enqueued` → synthesis runs
6. `mood_synthesis_waiting_for_sleep` should fire if (and only if) the photo handler reached `maybeFireSynthesis` before sleep was in.

### Risk

Low. The change shrinks logic rather than adding it. The keyboards being unconditional means a small additional friction — user always sees activities and photo prompts even on a duplicate run — but that was Roma's stated preference and matches the keyboard semantics (review and modify, not skip-if-present).

If future use shows the activities preamble is misleading or the keyboard pre-check state isn't obvious, that's a UX iteration on the keyboard, not on the flow router.

## 2026-05-06 — Mood flow Round 3: stripped mid-flow AI, end-only synthesis on Cloudflare-first cascade

### What

Live test on 2026-05-06 13:05 showed the mood synthesis hanging 127s+128s+128s on Gemini Pro 524s, retrying 3x then falling through to fixed text. Round 1 (queue migration) and Round 2 (stage tracking) had landed but couldn't help — the upstream Pro hangs were the bottleneck.

Round 3 redesign rebuilt the mood flow around two principles: **AI only at the end**, and **Cloudflare-first** with Gemini as fallback.

### Decision: end-only synthesis with day-level completeness

The entire mid-flow AI surface is gone:
- Score-tap acknowledgement: was Pro+thinking (10-25s, 127s on bad days). Replaced with a 2-tier Cloudflare-only micro-ack (Llama 3.3 70B fp8-fast → Gemma 3 12B → static text). 1-3s realistic latency.
- Emotions-Done acknowledgement: was Pro+high-thinking via the queue. Replaced with the same 2-tier micro-ack.
- Activities-Done kick: was a synthetic `handleMessage` call to nudge the AI to call the next tool. Replaced with `driveNextDeterministicStep()` which reads what's missing from D1 and sends the right keyboard or prose ask. Zero AI involved in flow routing.
- Photo-skip kick: same as activities-done.

End-of-flow synthesis is now the ONLY heavy AI call per check-in. Fired exactly once when ALL pieces are in: score, emotions, activities, sleep, photo-or-skipped — all checked across ALL of today's mood rows (morning + midday + evening combined). The cascade is 6 tiers:

1. `@cf/meta/llama-3.3-70b-instruct-fp8-fast` (15s)
2. `@cf/google/gemma-3-12b-it` (15s)
3. `gemini-3.1-flash-lite-preview` (20s)
4. `gemini-3-flash-preview` (25s)
5. `gemini-3.1-pro-preview` (30s)
6. Fixed warm fallback text

Mixed error-feedback + timeout per tier. Each tier wrapped in `Promise.race` so silent hangs (the 524 pattern) bail at the timeout cap. Realistic case: Tier 1 succeeds in 5-8s. Worst case before fixed text: 105s.

### Why "day-level" matters

The day-level completeness check means if sleep was logged in the morning entry (via `log_mood_entry` from a chat reply), the evening flow doesn't ask again. Same for emotions and photo. The synthesis sees the combined picture of the day, not just the evening row. Implemented via `moodStore.getDayEntries(env, userId, date)` which returns all rows for the date; `computeDayCompleteness()` walks them and OR's each piece.

### Prompt style: situation + role, no instructions

All three new prompts (score-ack, emotions-ack, synthesis) use the "describe the situation, set the role, get out of the way" style. No "max N words", no "do NOT mention X", no examples. Just data + "Briefly respond as a supportive and understanding friend who notices." The persona system prompt handles tone and severity.

Corrections applied during prompt design:
- Full poll option text in the score scale (not abbreviated buckets), so the model sees the same wording the user did.
- Dynamic `${period}` ("this morning" / "this afternoon" / "this evening" / "tonight") computed from `flow.started_at` via new `getCheckinTiming()` helper in `moodFlow.js`. Works for cron-triggered AND manual `/mood` at any time of day.
- Removed the word "clinical" from prompts — "His patterns and known triggers from past sessions" reads more like a friend's recall than a chart review.
- Helpline trio (Samaritans 116 123, SHOUT text 85258, NHS 111) only in the end-of-flow synthesis prompt, conditional on safety concern in the data.

### What changed

**New files:**
- `src/services/moodMicroAck.js` (~280 lines) — score-ack + emotions-ack runners, 2-tier Cloudflare cascade, mixed error+timeout, prompt builders.
- `src/services/moodSynthesis.js` (~330 lines) — 6-tier cascade, day-level completeness checker, `maybeFireSynthesis()` helper with idempotency guard (`mood_synthesis_fired_${chatId}_${date}` KV key, 26h TTL).

**Modified:**
- `src/lib/moodFlow.js` — added `getCheckinTiming(startedAtMs)` helper. Pure function, no side effects.
- `src/index.js` — stripped 100+ line AI block from `handleMoodPollAnswer`, replaced with `runScoreAck()` call. Deleted `mood_synthesis_after_emotions` and `mood_kick_after_emotions` queue cases. Added `mood_synthesis_final` queue case with looped typing indicator (every 4s during synthesis).
- `src/bot/handlers.js` — stripped queue-based AI synthesis from `mood_emo_done`, replaced with `runEmotionsAck()` + `driveNextDeterministicStep()`. Activities-Done and photo-skip handlers replaced their `handleMessage` kicks with `driveNextDeterministicStep()`. Photo upload handler now calls `maybeFireSynthesis()` and returns early so the AI generation pipeline doesn't run on top.
- `src/tools/mood.js` — `logMoodEntryTool.execute` now calls `maybeFireSynthesis()` after `advanceStage()`. Returns a new `next_step` value when synthesis fired so the AI knows to send a brief one-line ack instead of a long wrap-up reply (the synthesis IS the wrap-up).

### What is NOT changed

- `getCheckinRoadmap` HARD RULES block in `handlers.js` — still there for the prose-driven sleep ask path. The deterministic flow now handles activities, photo, and the post-emotions transition; sleep remains a prose ask answered by `log_mood_entry`.
- Safety net (`mood_flow_safety_net`) queue case — still works the same way, still uses stage values, still picks keyboards based on what's missing.
- The 524 streaming-path retryable error issue in handlers.js ~line 2050 — now moot for mood flows (we don't call Pro mid-flow at all). Non-mood Pro paths still hit it; revisit if logs show a problem.

### Risk acknowledged

If the entire 6-tier synthesis cascade fails, the user gets a fixed warm text and zero personalised AI engagement that day. That's a real UX regression compared to the previous setup's "something" — but the previous "something" was often itself broken (today's 127s×3 retries produced no synthesis either). 6 tiers makes total cascade failure rare.

The micro-acks have only 2 tiers and no Gemini fallback. If Workers AI is down, the score-tap and emotions-Done both fall back to short static text ("Got it. Tap below to share what you are feeling." and "Got it."). Mid-flow micro-acks can fail without the user noticing much; the synthesis is what carries the weight.

### Verification plan

1. `npx wrangler deploy --dry-run` to typecheck the new modules and dynamic imports.
2. `npx wrangler deploy --minify` to ship.
3. `/mood` end-to-end:
   - Tap score → confirm 1-3s ack arrives.
   - Tap Positive → select emotions → Done → confirm 1-3s ack arrives, activities keyboard sent immediately after.
   - Tap activities → Done → confirm photo keyboard sent immediately.
   - Upload photo (or Skip) → confirm sleep ask sent immediately.
   - Reply with "7 hours" → confirm log_mood_entry runs, brief one-line ack from AI, then the synthesis arrives 5-15s later (typing indicator visible throughout).
4. Tail logs to confirm:
   - No `mood_synthesis_after_emotions` or `mood_kick_after_emotions` queue events.
   - Single `mood_synthesis_enqueued` per check-in.
   - `mood_synthesis_tier_ok` showing which tier succeeded.
   - No `mood_synthesis_cascade_exhausted` under normal conditions.
5. Day-level test: log sleep in the morning (via chat), trigger evening `/mood`, confirm sleep is NOT re-asked at the end of the flow.

### Open thread

None for mood flow. Round 3 closes the loop.

The 524 fix in `handlers.js` ~line 2050 (Bug 4) is now lower-priority — mood flow no longer touches Gemini Pro mid-flow. Revisit only if non-mood Pro paths show similar hangs in tail logs.

## 2026-05-06 — Mood flow stage tracking: Option B "what's next missing"

### What

Live test on 2026-05-06 12:09 fired the mood-flow safety net with `stage:
"score"` despite the user having completed the score AND emotions AND clicked
Done ~18 minutes earlier. Safety net rendered the activities keyboard
correctly (its branches handle missing pieces deterministically) but only
because it fell through the unrecognised 'score' branch into the
activities-missing fallback. The stage value was lying about reality.

The earlier patch round had moved the synthesis chain into the queue with two
decoupled steps (`mood_synthesis_after_emotions`, `mood_kick_after_emotions`)
and thinned the `mood_emo_done` callback from 110 lines to ~30. That fixed
the waitUntil exhaustion. It did NOT fix the stage tracking, because no call
site ever advanced the stage when score/emotions/activities/sleep landed in
D1. Stage was only set in three places: `startFlow` (→'score'),
`initActivitiesPending` (→'activities'), `initPhotoPending` (→'photo'), and a
hardcoded `setStage(…'wrap')` inside the photo upload handler.

### Decision: Option B (stage = "what's next missing")

Original proposal was Option A: add `_done` markers like 'emotions_done',
'activities_done', so the safety net could read "past emotions" and pick the
right recovery. Pushed back — doubles the enum, and `emotions_done` is a
transient state that the safety net would still need to interpret as "now
render activities". The interpretation belongs at the transition point, not
in the safety net.

Option B: stage value at any moment IS the next missing piece. Canonical
order: emotions → activities → photo → sleep → wrap. Whenever a piece
lands, re-derive the stage from D1 + flow state. Safety net branches
already match these names exactly — zero changes needed downstream.

Second decision point: where to fire the advance for the AI-driven sleep
case (the model calls `log_mood_entry` with sleep_hours after asking in
prose). Option (i): inside `log_mood_entry` tool's execute function.
Option (ii): a check at the start of every `handleMessage`. Picked (i) —
the tool owns the state advancement when it writes the data, no wasteful
checks on every turn.

### What changed

**`src/lib/moodFlow.js`** — added `import * as moodStore` and new function
`advanceStage(env, chatId, userId)`. Reads today's evening entry from D1,
checks emotions/activities/photo/sleep presence, and sets `stage` to the
first missing piece in canonical order. Idempotent. ~50 lines including
docs that list every call site.

**`src/index.js`** — one call site in `handleMoodPollAnswer` after the score
upsert. After this, `stage` will be 'emotions' when the emotion buttons are
shown.

**`src/bot/handlers.js`** — four call sites:
- `mood_emo_done` callback after the emotions upsert.
- `mood_act|done` handler after `clearActivitiesPending`.
- Photo upload handler in `handleMessage`. Replaced hardcoded
  `setStage(env, chatId, 'wrap')` with `advanceStage(...)`. **Semantic
  change**: if a user uploads a photo before logging sleep, stage will now be
  'sleep' not 'wrap'. That's the correct behaviour — the safety net should
  ask for sleep, not wrap up early.
- `handlePhotoSkipCallback` — added `userId` lookup and `advanceStage` call
  after `markPhotoSkipped`. Previously this function did not advance the
  stage at all, leaving it at 'photo' even after skip.

**`src/tools/mood.js`** — added `import * as moodFlow` and gated
`advanceStage` call after `upsertEntry` inside `logMoodEntryTool.execute`.
Guarded on `context?.chatId`. Catch-all error handler so a flow miss never
breaks a tool call.

Every call site has a `.catch((e) => log.warn('mood_flow_advance_failed',
{...}))` so partial failure is observable but never throws. Where-tag in
logs identifies which call site failed: `after_score`, `after_emotions`,
`after_activities`, `after_photo_upload`, `after_photo_skip`, plus the
bare-catch in the tool path.

### What is NOT changed

- Safety net branches in `src/index.js` — already match the new stage names.
- The 'score' initial stage in `startFlow` — technically dead now (overwritten
  the moment score is logged) but harmless and matches the historical shape.
- Roadmap builder `getCheckinRoadmap` — reads from D1 directly, doesn't care
  about flow state.
- The unrelated 524 cascade and chain-fallback work — still pending, separate
  patch.

### Verification plan

1. Typecheck via `npx wrangler deploy --dry-run` (catches missing imports).
2. Deploy via `npx wrangler deploy --minify`.
3. Run a fresh `/mood` end-to-end: tap score → tap Positive → select an
   emotion → tap Done. Watch tail for absence of `mood_flow_advance_failed`
   warnings and absence of `queue_safety_net_done` (flow should make
   progress, never stall).
4. Stall recovery: trigger an evening flow, complete some pieces partially,
   then ignore for 20 minutes. Safety net should fire with the correct
   stage matching what's actually missing in D1.
5. Sleep advancement: when AI calls `log_mood_entry` with sleep_hours,
   confirm flow stage moves past 'sleep' to next missing or 'wrap'.

### Open thread

524-cascade fix and chain-fallback (Pro→Flash→Flash-Lite→Gemma) still
pending. Stage tracking is independent of those — it makes the safety net
resilient regardless of which model is doing the upstream work.

## 2026-05-05 (latest) — Mood flow tool-use bug: prompt fix attempt (Option 1)

### What

First live test of the AI-driven evening flow (the hybrid Option A+C from the
entry below) failed at the activities step. After `mood_synthesis_sent`, the
next user message routed to `gemini-3.1-flash-lite-preview` with
`route_reason: "simple_message"`. The model **ignored** the
`getCheckinRoadmap` instruction to call `send_activities_keyboard` and produced
a 66-char generic text reply with `tool_calls: 0`. No errors — a clean
success from the SDK's perspective. So the fallback chain in `c58d0f4` (which
is error-gated) never fired.

### Diagnosis (separated facts vs interpretation)

**Facts:**
- Flash-Lite supports tool use per Google's docs (Vertex AI + ai.google.dev),
  and the Gemini CLI uses it as a routing classifier in production.
- The router rule that picks Pro for `healthCheckinActive` requires the
  message to ALSO match the emotional regex. "7" or "a few hours" does not.
- The roadmap text was buried inside dynamicContext, after ~22KB of persona,
  style card, memory, curator block, and other context.
- Wording was soft ("ALWAYS prefer calling X over asking in prose") rather than
  hard ("call X. Do NOT ask in prose.").

**Interpretation (what we don't know for certain):**
- Whether the failure was caused by (a) prompt position, (b) prompt wording,
  (c) Flash-Lite's instruction-following weakness on long prompts, or (d) the
  combination. We have one sample, not a distribution.

### Options considered

1. **Strengthen the prompt only** (chosen). Cheapest test. No router change,
   no new fallback machinery, no behavioural risk.
2. Add a quality-signal fallback: when check-in is active and Flash-Lite
   responds with 0 tool calls + short text, retry on Flash. Real change but
   narrowly scoped.
3. Patch router: route active check-ins to Flash regardless of emotional
   content. Costs slightly more per check-in turn.

Roma's call: option 1 first, then escalate if it still fails.

### What changed (commit pending)

Replaced `getCheckinRoadmap` in `src/bot/handlers.js` (~line 268). New version:

- Builds an **ordered** list of missing pieces: activities → sleep → photo.
  Mood score and emotions are listed as `preCaptured` (state info, not
  actionable), since they are poll/button-driven.
- Surfaces the **single next required action** by name, not a list of
  everything missing. This avoids the model trying to chain tool calls in one
  turn (which interacts badly with `MAX_PASSES = 4`).
- Wraps the directive in clearly delimited ALL-CAPS markers:
  `=== EVENING CHECK-IN — HARD RULES (override anything else in this prompt) ===`
- Imperative, tool-name-explicit wording: "NEXT REQUIRED ACTION: call
  send_activities_keyboard. Do NOT ask about activities in prose. The tool is
  the answer."
- Five numbered rules: drop check-in if user changes topic; do exactly ONE
  thing per turn; tool IS the question for activities/photo; sleep is
  prose-then-tool; commit_journal_entry then wrap-up at the end.

Dropped the "any final thoughts or notes" item (formerly required `entry.note`)
because it was unbounded — it would always block `commit_journal_entry`. The
wrap-up message naturally invites further conversation.

No router change. No callback-handler change. No fallback machinery added.

### How to test

```
cd ~/Documents/GitHub/gemini-bot
npx wrangler deploy --minify
```

Then `/mood` → poll → emotions → Done. Watch `wrangler tail` for:

| Signal                                                            | Pass condition                                |
|-------------------------------------------------------------------|-----------------------------------------------|
| `tool_calls > 0` and `tool_names` includes `send_activities_keyboard` on the post-emotions turn | Patch worked                                  |
| `max_passes_hit` events                                           | Should stay at 0 — model isn't chaining tools |

If `tool_calls: 0` again, position outweighs wording at this depth, and we
escalate to option 2 (quality-signal fallback) or option 3 (router patch).

### Risks acknowledged

- **Position vs wording:** if it's truly position, no amount of wording fixes
  it. We will know after one or two real check-ins.
- **Cache churn:** `dynamicContext` already changes per turn (memory counts,
  remaining items, etc.), so the new wording adds no new cache pressure. The
  earlier worry was unfounded.
- **Pro over-calling:** rules are state-gated by D1 reads, not model judgment.
  The model can only call a tool the roadmap says is missing. State filtering
  prevents hallucinated tool calls.
- **Pass-cap interaction:** by surfacing only the NEXT action (not all missing
  pieces), we explicitly discourage chaining. `MAX_PASSES = 4` should hold.

## 2026-05-05 (latest+1) — Mood flow: emotions-done handler doesn't kick the AI

### What

The HARD RULES roadmap patch (entry above) was fundamentally sound but
targeted the wrong layer. After clicking Done on the emotions buttons:

1. `mood_emo_done` callback fires → logs emotions, runs Pro+high synthesis,
   sends synthesis text. ✅ working.
2. Handler then deletes `health_checkin_active_${chatId}` and returns. ⛔
3. User reads the synthesis. They do not type anything because the bot's
   message reads as a complete therapeutic reply, not an open question.
4. Without a user message, `handleMessage` never runs, so the roadmap never
   gets injected, so the AI never picks up the flow. The check-in dies
   silently after the synthesis.

The AI-driven design assumed *something* would invoke `handleMessage` after
the synthesis. That "something" was meant to be the user's reply — but in
practice the synthesis already feels like the natural end of the exchange.

### What changed

Two edits inside the `mood_emo_done` handler in `src/bot/handlers.js` (~line 2598):

1. **Removed** the `await env.CHAT_KV.delete('health_checkin_active_${chatId}')`
   line. The flag must survive past synthesis so the next handler invocation
   sees an active check-in. `commit_journal_entry` now exclusively owns the
   flag's lifecycle (this matches the original intent in the previous
   session's design but had been broken by a stale early-clear).
2. **Added** a synthetic-message kick after `mood_synthesis_sent` logs.
   Constructs a fake msg object with text
   `[Emotion selection complete. Continue the check-in by performing the
   NEXT REQUIRED ACTION from the HARD RULES roadmap.]` and awaits
   `handleMessage(kickMsg, env)`. The kick lands in the full handler pipeline
   (tagger → memory → router → generation), which now sees an active
   check-in flag and applies the patched roadmap with NEXT REQUIRED ACTION =
   `send_activities_keyboard`.

### Risks acknowledged before deploy

- **Bracketed text and `stripLeakedThoughts`:** the kick text uses square
  brackets, but the strip regex only matches a closed list of verbs
  (`Noticing`, `Thinking`, etc.) — "Emotion" is not in the list, so it
  survives. Verified.
- **Latency:** the kick runs through the full pipeline (~5-10s) so the
  activities keyboard appears a few seconds after the synthesis. Acceptable

## 2026-05-06 (latest+2) — Mood flow Plan A: full queue migration

### What

The kick patch (entry above) was correct in mechanism but doomed by
architecture. Two heavy generations chained back-to-back inside one
`waitUntil` window cannot fit the 30s post-response budget:

- Pro+high-thinking synthesis: 10-25s
- Full `handleMessage` pipeline (tagger, memory, route, cache, generation):
  another 5-15s
- Telegram callback responded in <1s, so the budget started counting from
  the moment the synthesis began

Result: synthesis sent, kick started, generation cancelled mid-stream.
User saw synthesis, then nothing.

Project notes already explicitly flag this constraint:

> `ctx.waitUntil` ceiling is 30 seconds post-response (not configurable
> `cpu_ms`). Media/async work must use Queues (15-min wall clock, 3 retries)
> for reliability.

I missed applying that pattern to the new emotion-flow chain and got
bitten by it. Roma asked for the queue migration explicitly: "all the
mood check steps should be on the queue".

### Architecture decision: Plan A (queue) vs Plan B (Cloudflare Workflow)

Weighed two architectures. Plan A: each heavy step becomes a `TASK_QUEUE`
message, matching the existing `mood_poll_answer` pattern. ~30 min of work,
low risk, no architectural lock-in. Plan B: full Cloudflare Workflow
rewrite (already on the project's roadmap as a dedicated session).

Chose A. Rationale:
- Bug is a `waitUntil` exhaustion; queue solves that. Workflow solves it
  AND many other things we didn't ask for.
- Mid-debug refactor is the wrong time to switch architectures.
- Workflow design needs answers to non-trivial questions (one workflow
  per check-in vs per user, how Telegram callbacks wake a running
  instance, state size, idempotency) — those belong in a dedicated
  design session, not a panic patch.
- Roma's project notes already define Workflows as "agreed for a
  dedicated session" — keeping that scope for later.

### What changed

**`src/bot/handlers.js` — `mood_emo_done` callback handler (~line 2486):**

From 110 lines (synthesis + history + kick) down to ~30 lines. Now does:
1. Edit message markup
2. Save emotions to D1 (`upsertEntry`)
3. Fire-and-forget tagger (CF AI, free, ~500ms)
4. Clear emo selected KV
5. Enqueue `mood_synthesis_after_emotions` to TASK_QUEUE
6. Return

Total inline time: <1s. No Pro calls, no `handleMessage`, nothing that
could stall the `waitUntil` budget.

Fallback path: if `env.TASK_QUEUE` is missing or send fails, post a brief
acknowledgement so the user isn't left staring at a dead chat. The deep
synthesis is sacrificed in that edge case but the flow continues.

**`src/index.js` — queue consumer (added 2 cases after `mood_poll_answer`):**

1. `mood_synthesis_after_emotions`:
   - Re-derives all context fresh from D1/KV/Vectorize (don't trust
     cached state — the queue could fire minutes later in theory).
     Pulls: today's entry, recent 30-day mood history, therapeutic notes,
     semantic vector context, relevant past episodes by emotion.
   - Builds the same therapeutic synthesis prompt as before.
   - Calls `generateDeepResponse` with `thinkingLevel: 'high'`,
     `maxOutputTokens: 2000` (Pro + high thinking, matching Eukara's pattern).
   - Sends synthesis to Telegram.
   - Persists synthetic user turn + synthesis in chat history KV.
   - Enqueues `mood_kick_after_emotions` as a SEPARATE message so the
     synthesis success can't be rolled back by a kick failure.
   - On final retry (attempt 3) failure: sends a brief acknowledgement so
     the user isn't hanging. Throws to retry queue otherwise.

2. `mood_kick_after_emotions`:
   - Builds the same synthetic kick message as before.
   - Awaits `handleMessage(kickMsg, env)` — fresh 15-min budget.
   - On failure: don't notify user (synthesis already landed). Re-throw
     for queue retry. After 3 retries the user can simply type to resume
     the flow naturally.

### Why two queue messages instead of one

Idempotency. If synthesis succeeds and kick fails, retrying ONLY the kick
is safe — synthesis was already sent to Telegram, no duplicate. If we put
both in one message, retrying after kick failure would re-run synthesis
and send a duplicate to the user.

Each message also gets its own 15-min budget and 3-retry budget. If Pro
is slow on synthesis (e.g. 4-min preview overload), the kick still has a
full 15 min when its turn comes.

### Risks acknowledged

- **Latency between synthesis and kick:** queue dispatch adds ~1-3s.
  Acceptable; user is reading the synthesis during this window.
- **Queue failure:** the inline fallback in `mood_emo_done` covers
  TASK_QUEUE missing or send failing. The synthesis fallback covers
  Pro generation failing. The kick has no user-facing fallback because
  the synthesis is already there — user can type to resume.
- **Context staleness:** queue context is re-derived fresh. The only
  thing passed via the queue message is `selected` (the emotion array)
  — everything else comes from D1/KV/Vectorize at queue-handler time.
- **Existing `mood_flow_safety_net` queue task is now redundant** for the
  immediate-after-emotions case (we no longer rely on the AI noticing the
  flow on its own). It still serves as a safety net for genuinely stalled
  flows (e.g. user starts then walks away). Leaving it in place.

## 2026-05-05 (later) — Evening mood check-in: source-based suppression + AI-driven flow with safety net

### What

Full redesign of the evening mood check-in. Two problems addressed in one pass:

1. **Cron suppression bug:** Last cron-shaped evening row was 30 Apr 22:50 UTC.
   After that, every "evening" row was created at non-cron times by mid-day mood
   mentions tagged via `log_mood_entry`. The cron's `hasCheckedInToday` saw any

### Problem 4: Telegram HTML parse failures from model-emitted unknown tags

User flagged the `<b>Evening Affirmations</b>` rendering and asked whether
formatting was Telegram-compatible end-to-end. Investigation also surfaced
the pre-existing `Bad Request: can't parse entities: Unsupported start tag
"string"` error from earlier in the day.

Root cause: `sanitizeTelegramHTML` in `src/lib/formatter.js` used a deny-list
of known-bad tags (`p`, `div`, `br`, `h1-6`, `span` without tg-spoiler, etc.).
Anything the model invented outside that list (`<string>`, `<example>`,
`<note>`, etc.) slipped through to a Telegram parse failure. The strip-all-tags
fallback in `telegram.sendMessage` caught it but users saw degraded output.

Reminder cron path in `src/index.js` already escapes user content correctly
via `escapeHtml` (added in commit `f465422`), so reminders themselves are
not the failure source. The failures come from bot-generated wrappers around
model output (e.g. `<b>Weekly Report</b>\n\n${report}`) where ${report}
contained unexpected tags.

Fix (Option A — allow-list rewrite). Commit `7c724d2`. Rewrote
`sanitizeTelegramHTML` in `src/lib/formatter.js`:
  1. Markdown pre-processing kept (`**bold**`, `__italic__` -> `<b>`/`<i>`)
  2. Allow-list strip: every tag matched against Telegram spec. Plain
     allow-list: b, strong, i, em, u, ins, s, strike, del, tg-spoiler,
     code, pre, blockquote. Attribute-bearing tags handled separately:
     `<a href>` (required href), `<span class=tg-spoiler>` (only that class),
     `<tg-emoji emoji-id=NNN>` (numeric only), `<blockquote expandable>`
     (expandable flag preserved)
  3. Existing balancer (orphan closers / unclosed openers) and whitespace
     collapse retained unchanged
  4. HTML entities (&lt; etc) left untouched so reminder cron's escapeHtml
     output stays escaped

Test coverage: 23 cases ran locally before deploy, all passed. Including the
real failing case (`<string>foo</string>` strips cleanly), pre-escaped
entity preservation, all spec-allowed attribute-bearing tags, balancer
correctness on orphans/unclosed.

Acknowledged trade-off: `<code class="language-python">` attributes are now
stripped. Syntax highlighting in Telegram code blocks no longer works.
Cosmetic only — parse still succeeds. If anyone needs language-hinted code
blocks back, add a special case for `<code>` matching the pre/code spec.

Deferred to a follow-up Option C session: centralised
`formatTelegramMessage(template, fields)` helper to make bot-template +
user-data interpolation auto-escape correctly. Currently five sites in
`src/index.js` (Weekly Report, Mid-Week Check-in, Monthly self-assessment,
Things I found interesting, Architecture Deep Search title) interpolate
model output directly into `<b>Title</b>\n\n${msg}` templates. The new
sanitiser catches recurrences but a centralised helper would make this
discipline impossible to forget.
